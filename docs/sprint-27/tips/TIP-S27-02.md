model SerialNumberingRule {
  id                 String        @id @default(cuid())

### 3) Numbering service — `src/lib/serial/numbering.ts`

**Signature:**
```ts
export interface GenerateSerialOptions {
  moduleDesignId: string;
  now?: Date;                  // inject cho test
  prismaClient?: PrismaClient; // inject cho test, default import global
}

export async function generateSerial(opts: GenerateSerialOptions): Promise<string>;
```

**Behavior (spec chi tiết — phải theo):**
1. Lookup `SerialNumberingRule` theo `moduleDesignId`. Nếu không có → throw `SerialNumberingRuleNotFoundError(moduleDesignId)`.
2. Tính `currentMMYY` từ `now` (format integer: tháng 9/2025 = 925, tháng 10/2025 = 1025 — tức `month*100 + year%100`. **Hoặc** dùng format MMYY 4-digit: "0925", "1025". Chọn format nào cũng được nhưng test phải phủ rollover. Khuyến nghị: integer MMYY `month*100 + (year%100)` vì so sánh dễ).
3. Atomic transaction (`prisma.$transaction` với isolation `Serializable` hoặc dùng `SELECT ... FOR UPDATE` qua raw query):
   - Re-read rule inside transaction.
   - Nếu `rule.counterLastMMYY !== currentMMYY` → reset `counter = 1`, `counterLastMMYY = currentMMYY`.
   - Else → `counter = rule.counter + 1`.
   - Update rule.
   - Format serial theo template: `{prefix}-{version}-{ddmmyy}-{###}` với `###` = zero-pad 3 digit (041, 100, 999, 1000 — nếu >999 thì không pad, cứ nguyên chiều dài).
   - `ddmmyy` = "091025" (dd=09, mm=10, yy=25). Tức day-month-year 2 digit mỗi phần.
   - Return serial string.
4. Nếu `$transaction` fail do conflict (P2034) → retry tối đa 3 lần với backoff 10-50ms random.
5. Nếu vẫn fail → throw `SerialGenerationConflictError`.

**Helper export:**
```ts
export function formatSerialDate(d: Date): string;  // "091025"
export function computeMMYY(d: Date): number;       // 1025
export class SerialNumberingRuleNotFoundError extends Error {}
export class SerialGenerationConflictError extends Error {}
```

### 4) API routes

Tất cả route wrap `requireRole(...)` + try/catch + Zod validate input.

**`POST /api/serial/generate`** — role: `engineer`, `production`, `admin`
- Body Zod: `{ moduleDesignId: z.string().cuid() }`
- Response 201: `{ serial: string, serialUnitId: string }`
- Behavior: call `generateSerial({ moduleDesignId })` → tạo `SerialUnit` mới với `source=MANUFACTURED`, `status=IN_STOCK`, `createdByUserId=session.user.id`, `moduleDesignId=body.moduleDesignId`. Return serial + id.
- Error: 404 nếu moduleDesignId không tồn tại, 422 nếu rule chưa setup.

**`GET /api/serial/:serial`** — role: tất cả authenticated (không role-gate để tra cứu rộng)
- Response 200: full SerialUnit + include `moduleDesign`, `product`, `part`, `inventory`, `warehouse`, `createdByUser (safe fields only)`, `parentLinks.parentSerial`, `childLinks.childSerial`.
- 404 nếu serial không tồn tại.

**`POST /api/serial/:serial/status`** — role: `warehouse`, `production`, `admin`
- Body Zod: `{ status: z.nativeEnum(SerialStatus), note: z.string().optional() }`
- Response 200: updated SerialUnit.
- Behavior: update `status` + append `note` vào `notes` (prepend timestamp + userId). Nếu chuyển sang `SHIPPED` hoặc `CONSUMED` thì record timestamp vào `meta.statusHistory[]`.
- Transition validation (soft): không cho `SHIPPED → IN_STOCK` trực tiếp (phải qua `RETURNED`). Ghi thành const `ALLOWED_TRANSITIONS` ở top file.
